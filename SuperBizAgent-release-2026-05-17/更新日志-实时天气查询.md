## 一、为什么要加天气查询？

### 背景：智能助手能力拓展

SuperBizAgent 作为专业的智能运维助手，除了处理告警、日志、文档检索等运维任务外，也需要覆盖日常办公场景中的通用信息查询需求。天气查询是最常见的通用查询之一：

- **运维场景关联**：极端天气可能导致 IDC 机房温控异常、网络中断等问题，快速获取天气信息有助于预判风险
- **日常办公辅助**：团队成员可以在工作流中直接询问天气，无需切换工具
- **Agent 能力完整性**：补齐时间（DateTimeTools）、空间（天气）、文档（RAG）、告警（Prometheus）四维信息获取能力

## 二、改了什么？

### 2.1 新建文件：`WeatherTools.java`

**路径**：`src/main/java/org/example/agent/tool/WeatherTools.java`

这是核心改动。它是一个 Spring `@Component`，封装了 **wttr.in** 免费天气 API 的调用。

#### API 详情

| 项目 | 值 |
|------|-----|
| **API 地址** | `https://wttr.in/{city}?format=j1&lang=zh` |
| **认证方式** | 无需 API Key（免费开放 API） |
| **超时** | 15 秒（连接 + 读取） |
| **数据格式** | JSON |
| **语言** | 中文（`lang=zh`） |

#### 核心方法

```java
@Tool(description = "Query real-time weather information for a specified city...")
public String queryWeather(String city)
```

- **参数** `city`：城市名称，支持中文（如"北京"、"上海"）或英文（如"Beijing"、"London"）
- **返回**：包含以下字段的 JSON 字符串

| 字段 | 说明 | 示例值 |
|------|------|--------|
| `success` | 查询是否成功 | `true` |
| `city` | 查询城市 | `北京` |
| `area_name` | 气象站地区名 | `Beijing` |
| `country` | 国家 | `中国` |
| `temperature` | 当前温度 | `28°C` |
| `feels_like` | 体感温度 | `30°C` |
| `max_temp` | 今日最高温 | `33°C` |
| `min_temp` | 今日最低温 | `22°C` |
| `humidity` | 湿度 | `65%` |
| `weather_desc` | 天气状况描述 | `晴` |
| `wind_speed` | 风速 | `15 km/h` |
| `wind_direction` | 风向 | `东南` |
| `visibility` | 能见度 | `10 km` |
| `pressure` | 气压 | `1013 hPa` |
| `cloud_cover` | 云量 | `25%` |
| `uv_index` | 紫外线指数 | `5` |
| `sunrise` | 日出时间 | `05:12 AM` |
| `sunset` | 日落时间 | `07:35 PM` |
| `message` | 状态信息 | `成功获取 北京 的实时天气数据` |

#### 数据提取逻辑

wttr.in 返回的原始 JSON 结构较复杂（多层嵌套、多语言描述），`WeatherTools` 内部做了以下处理：

1. **当前天气状况** (`current_condition`)：提取温度、体感温度、湿度、天气描述、风速风向、能见度、气压、云量、紫外线指数
2. **今日预报** (`weather[0]`)：提取最高/最低温度
3. **天文信息** (`astronomy`)：提取日出/日落时间
4. **最近气象站** (`nearest_area`)：提取气象站名称和国家

所有字段统一封装为 `SimplifiedWeather` 结构返回给 LLM，避免原始 JSON 的冗余信息干扰模型判断。

#### 错误处理

- **城市名称为空** → 返回 `{"success": false, "message": "城市名称不能为空"}`
- **API 请求失败** → 返回 HTTP 状态码和错误信息
- **城市未找到** → 返回 `{"success": false, "message": "城市未找到，请检查城市名称是否正确"}`
- **数据解析异常** → 返回解析错误详情

## 三、架构说明

### 工具链路

```
用户提问（"北京今天天气怎么样？"）
  → ChatController (/api/chat 或 /api/chat_stream)
  → ChatService.createReactAgent() 构建 Agent
  → ReactAgent 识别天气查询意图 → 自动调用 WeatherTools.queryWeather("北京")

  → ReactAgent 将天气数据组织成自然语言回复
  → 返回用户
```

## 五、技术选型说明

### 为什么选择 wttr.in 而不是 OpenWeatherMap / 和风天气？

| 维度 | wttr.in | OpenWeatherMap | 和风天气 |
|------|---------|----------------|----------|
| **API Key** | ❌ 不需要 | ✅ 需要注册 | ✅ 需要注册 |
| **免费额度** | 无限制 | 1000次/天 | 1000次/天 |
| **中文支持** | ✅ 支持 | 需付费版 | ✅ 原生 |
| **城市查询** | 中/英文 | 英文为主 | 中文/ID |
| **数据完整度** | 中等 | 高 | 高 |
| **部署复杂度** | 零配置 | 需管理密钥 | 需管理密钥 |

选择 **wttr.in** 的理由：
1. **零配置** — 无需注册、无需 API Key，即装即用
2. **免费用量** — 无调用次数限制，适合内部工具场景
3. **全球化** — 支持全球所有城市，中英文混合查询
4. **数据够用** — 温度、体感温度、湿度、风速、日出日落等核心信息齐全

若未来需要更高精度的天气数据（如逐小时预报、空气质量指数 AQI），可升级为和风天气 API（中国城市数据最完整）或 OpenWeatherMap One Call API。

## 六、Bug 修复记录

### 2026-06-10：Jackson 反序列化兼容性修复

**问题**：wttr.in 返回的 JSON 中包含未映射字段（如 `FeelsLikeF`、`temp_F`、`windspeedMiles` 等英制单位字段），导致 Jackson 反序列化抛出 `UnrecognizedPropertyException`。

**根因**：Jackson 默认的 `ObjectMapper` 在遇到 Java 类中未声明的 JSON 字段时会抛出异常。

**修复**（双重保障）：

1. **ObjectMapper 全局配置** — 在字段声明处直接配置：
   ```java
   private final ObjectMapper objectMapper = new ObjectMapper()
           .configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);
   ```

2. **类级别注解** — 所有 wttr.in 数据模型类添加：
   ```java
   @JsonIgnoreProperties(ignoreUnknown = true)
   ```
   覆盖 `WttrResponse`、`WttrCurrentCondition`、`WttrNearestArea`、`WttrWeatherDay`、`WttrAstronomy`、`WttrValue` 共 6 个类。
