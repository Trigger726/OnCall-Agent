# 更新日志 — 检索重排功能（Rerank）

> 日期：2026-06-07
> 作者：Claude
> 功能：在向量检索召回后，集成阿里云百炼 Rerank API 对文档进行语义重排序

---

## 一、为什么要加重排？

### 问题：向量检索不够精准

原先的检索流程只有一步：

```
用户问题 → 生成向量(Embedding) → Milvus 向量搜索(L2距离) → 返回结果给 LLM
```

Milvus 使用 **L2 欧氏距离** 来衡量向量相似度，但 Embedding 模型生成的向量是**语义向量**，L2 距离并不总是能准确反映两段文字在"用户想表达的意思"上的相关性。

比如用户问"CPU 飙高怎么排查"，Milvus 可能把一篇讲"CPU 架构设计"的文档排到第 1 位（因为向量空间里它离得近），而真正有用的"CPU 故障排查手册"排在第 3 位。

### 解决：加入语义重排（Rerank）

新流程：

```
用户问题 → 生成向量(Embedding) → Milvus 向量召回(topK=3)
         → 阿里云百炼 gte-rerank 语义重排 → 返回重排结果给 LLM
```

**重排模型（gte-rerank）** 是一个专门为"给定一个问题，判断哪篇文档最相关"训练的模型。它会逐字逐句阅读每篇文档和问题，给出真实的相关性分数，而不是依赖向量空间的近似。

## 二、改了什么？

### 2.1 新建文件：`RerankService.java`

**路径**：`src/main/java/org/example/service/RerankService.java`

这是核心改动。它是一个 Spring `@Service`，封装了阿里云百炼 DashScope Rerank API 的调用。

#### API 详情

| 项目 | 值 |
|------|-----|
| **API 地址** | `https://dashscope.aliyuncs.com/api/v1/services/rerank/text-rerank/text-rerank` |
| **模型** | `gte-rerank`（阿里云百炼语义排序模型） |
| **认证方式** | `Authorization: Bearer {api_key}`（复用你已有的 DashScope API Key） |
| **不降级策略** | API 调用失败直接抛 `RuntimeException`，不返回未重排的结果 |

#### 核心方法

```java
// 方法 1：对任意文档列表进行重排
public List<RerankResult> rerank(String query, List<String> documents)
// RerankResult 包含：
//   - index:  原始文档在输入列表中的位置
//   - relevance_score: 语义相关性分数（0~1，越高越相关）

// 方法 2：专门处理 VectorSearchService.SearchResult 的重排
public List<SearchResult> rerankSearchResults(
    String query, List<SearchResult> searchResults)
// 这个方法会自动：
//   1. 从 SearchResult 中提取 content 字段
//   2. 调用 rerank() API
//   3. 用相关性分数替换原始的 L2 距离分数
//   4. 返回按相关性降序排列的新列表
```

#### 请求/响应示例

**发送给百炼 API 的请求**：
```json
{
  "model": "gte-rerank",
  "input": {
    "query": "CPU 飙高怎么排查",
    "documents": [
      "CPU 故障排查手册：第一步检查...",
      "CPU 架构设计原理...",
      "服务器 CPU 监控指标说明..."
    ]
  },
  "parameters": {
    "top_n": 3,
    "return_documents": false
  }
}
```

**百炼 API 返回**：
```json
{
  "output": {
    "results": [
      { "index": 0, "relevance_score": 0.92 },
      { "index": 2, "relevance_score": 0.78 },
      { "index": 1, "relevance_score": 0.45 }
    ]
  },
  "usage": { "total_tokens": 512 }
}
```

这表示：原始第 0 篇文档最相关（0.92 分），第 2 篇其次，第 1 篇最不相关。结果按分数降序重排。

### 2.2 修改文件：`application.yml`

**路径**：`src/main/resources/application.yml`

新增了 4 行配置：

```yaml
# 重排配置（阿里云百炼 DashScope Rerank API）
rerank:
  enabled: true        # 是否启用重排（false 则跳过，保持原顺序）
  model: gte-rerank    # 重排模型
  top-n: 3             # 重排后保留的最相关文档数量
```

你可以随时把 `enabled` 改成 `false` 来临时关闭重排，方便对比效果。

### 2.3 修改文件：`InternalDocsTools.java`

**路径**：`src/main/java/org/example/agent/tool/InternalDocsTools.java`

改动点：
1. **新增依赖注入**：构造函数增加 `RerankService` 参数
2. **新增 import**：`import org.example.service.RerankService;`
3. **检索流程改为两步**：

   ```java
   // 步骤 1：向量召回
   List<SearchResult> searchResults = vectorSearchService.searchSimilarDocuments(query, topK);
   
   // 步骤 2：语义重排（新增！）
   List<SearchResult> rerankedResults = rerankService.rerankSearchResults(query, searchResults);
   
   // 返回的是重排后的结果，分数也从 L2 距离变成了语义相关性分数
   return objectMapper.writeValueAsString(rerankedResults);
   ```

## 三、完整检索链路（改动后）

```
用户提问 "CPU 飙高怎么排查"
    │
    ▼
InternalDocsTools.queryInternalDocs(query)
    │
    ├─ 步骤 1: 向量召回
    │    VectorSearchService.searchSimilarDocuments(query, topK=3)
    │    ├─ VectorEmbeddingService.generateQueryVector(query)
    │    │    └─ DashScope text-embedding-v4 API
    │    └─ Milvus.search(向量, topK=3, metric=L2)
    │    返回: [doc_A(score=0.12), doc_B(score=0.18), doc_C(score=0.25)]
    │         （L2 距离越小越相似，但可能不精准）
    │
    ├─ 步骤 2: 语义重排（新增！）
    │    RerankService.rerankSearchResults(query, searchResults)
    │    ├─ 提取 3 篇文档的 content
    │    └─ DashScope gte-rerank API
    │        POST /api/v1/services/rerank/text-rerank/text-rerank
    │        请求: { model: "gte-rerank", query: "CPU飙高...", documents: [...] }
    │        返回: [{index:2, score:0.95}, {index:0, score:0.72}, {index:1, score:0.31}]
    │    重排后: [doc_C(score=0.95), doc_A(score=0.72), doc_B(score=0.31)]
    │         （按语义相关性降序，分数越高越相关）
    │
    ▼
返回重排后的 JSON 给 LLM（ReactAgent）
```

## 五、如何验证功能正常

1. **启动服务**：
   ```bash
   make init
   ```

2. **观察日志**，应该能看到：
   ```
   INFO  RerankService - 重排服务初始化完成 — enabled: true, model: gte-rerank, topN: 3
   ```

3. **发起一次文档查询**（比如在聊天界面问一个有知识库答案的问题），观察日志：
   ```
   INFO  InternalDocsTools - 向量召回完成，获取到 3 个候选文档，开始重排序...
   INFO  RerankService - 开始调用 DashScope Rerank API — query: ..., documents count: 3
   INFO  RerankService - 重排完成 — 返回 3 个结果
   INFO  RerankService -   #1: index=2, relevance_score=0.95
   INFO  RerankService -   #2: index=0, relevance_score=0.72
   INFO  RerankService -   #3: index=1, relevance_score=0.31
   INFO  InternalDocsTools - 重排序完成，最终返回 3 个文档
   ```

4. **对比测试**：在 `application.yml` 中设置 `rerank.enabled: false`，重启后问同一个问题，看返回的文档顺序和相关性是否有差异。

## 六、不降级策略说明

RerankService 设计为**不降级**：
- 如果 API 调用失败（网络错误、认证失败、返回异常等），直接抛出 `RuntimeException`
- 这么做是为了确保 LLM 拿到的文档顺序是最优的
- 如果降级返回原始排序，一旦向量检索不够准，LLM 可能基于不相关的文档给出错误答案

这区别于一般的"优雅降级"设计，是一个有意的工程选择。
