# 更新日志 — 智能助手基础功能完善 & 公司告警手册

> 日期：2026-06-08
> 作者：Claude
> 功能：补齐智能OnCall助手的核心交互能力、全面美化前端界面、新增公司级告警手册系统

---

## 一、为什么要做这些改动？

### 问题 1：Agent 基础交互能力不足

原版智能助手只能发送消息和切换快速/流式模式，缺少日常使用中最基本的操作：

- ❌ 无法**复制** AI 的回答
- ❌ 无法**重新生成**不满意的回答
- ❌ 无法**清空**当前对话（只能新建）
- ❌ 无法**导出**对话记录
- ❌ 没有**快捷提问**入口，用户不知道能问什么
- ❌ 没有**暗色模式**，夜间使用刺眼

### 问题 2：缺少公司告警知识体系

AI Ops 智能运维功能可以自动拉取 Prometheus 告警并分析，但缺少一个**静态的告警知识库**（告警手册），供值班SRE在收到告警时快速查阅：

- 这个告警是什么？
- 可能的原因有哪些？
- 我应该怎么排查？
- 处理不了时怎么升级？

这些是 OnCall 场景中最基础的需求。

### 问题 3：前端界面需要现代化

原版 UI 功能完整但视觉风格偏基础，需要提升到生产级水准。

## 二、改了什么？

### 2.1 公司告警手册（全新模块）

这是本次最大的新增功能，包含 **后端 API + 前端面板 + 15 条告警规则**。

#### 新建文件：`AlertRule.java`

**路径**：`src/main/java/org/example/dto/AlertRule.java`

告警规则的数据模型，包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | String | 告警编号（如 ALT-001） |
| `name` | String | 告警名称 |
| `category` | String | 分类：资源/服务/性能/数据库/中间件/安全/网络/容器/任务/数据 |
| `severity` | String | 严重级别：critical / warning / info |
| `description` | String | 告警描述 |
| `possibleCauses` | List\<String\> | 可能原因列表 |
| `diagnosticSteps` | List\<String\> | 排查步骤列表 |
| `resolutionSteps` | List\<String\> | 处理方案列表 |
| `escalationRule` | String | 升级规则 |
| `relatedDocs` | List\<String\> | 关联的内部文档文件名 |

#### 新建文件：`AlertHandbookController.java`

**路径**：`src/main/java/org/example/controller/AlertHandbookController.java`

提供 8 个 REST API 端点：

```
GET  /api/alert-handbook/all                          → 获取全部告警规则
GET  /api/alert-handbook/by-severity/{severity}       → 按严重级别筛选
GET  /api/alert-handbook/by-category/{category}       → 按分类筛选
GET  /api/alert-handbook/search?q={keyword}           → 关键词搜索
GET  /api/alert-handbook/detail/{id}                  → 获取单条详情
GET  /api/alert-handbook/categories                   → 获取所有分类
GET  /api/alert-handbook/summary                      → 获取统计摘要
POST /api/alert-handbook/reload                       → 热重载手册数据
```

数据从 `alert-handbook.json` 文件加载，支持热重载，无需重启服务即可更新告警规则。

```json
}
```

返回服务端当前所有活跃会话的信息，方便运维排查问题。

## 七、设计决策说明

### 为什么告警手册用静态 JSON 而不是数据库？

1. **运维友好**：JSON 文件可以直接用文本编辑器修改，通过 Git 做版本管理
2. **零依赖**：不需要额外的数据库表或迁移脚本
3. **热重载**：`/reload` 端点支持不重启更新
4. **可移植**：规则数据可以轻松复制到其他环境

后续如果需要动态管理（增删改 + 审批流程），可以迁移到数据库 + 管理后台。

### 为什么亮色/暗色模式用 CSS 变量而不是两套 CSS？

1. **维护成本低**：只需定义一套变量，修改时改一处即可
2. **切换性能好**：只需修改 `<html>` 的 `data-theme` 属性
3. **一致性好**：保证两种主题使用完全相同的布局和间距
